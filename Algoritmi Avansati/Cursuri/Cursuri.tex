P (Polynomial) - Clasa de probleme pentru care le putem afla solutia in timp polinomial \\
NP (Nondeterministic Polynomial) - Clasa de probleme pentru care putem verifica in timp polinomial daca un rezultat este solutie corecta pentru problema noastra. \\
Problema de Optim: \\
Informal spus este problema in care trebuie sa gasesti o “cea mai bună” solutie/constructie fezabila. \\
“Cea mai bună” - poate avea doua sensuri:   \\
\begin{itemize}
\item Fie avem o problema de minimizare precum Problema Comis-voiajorului. \\
\item Fie o problema de maximizare precum cea de a găsi o acoperire de cardinal maxim pentru multimea varfurilor unui graf \\
\end{itemize}
Fie P - o problema de optim, și I o intrare pe aceasta problema. Vom nota cu OPT(I) ”valoarea” soluției optime. \\
În mod analog, atunci când propunem un algoritm care să ofere o soluție fezabilă pentru problema
noastră, vom nota ”valoarea” acelei soluții cu ALG(I). \\
De cele mai multe ori, atunci când nu se creează confuzie, vom simplifica notațiile folosind termenii
”OPT”, respectiv ”ALG” \\
Pe parcursul prezentării vom presupune că atât OPT, cât și ALG sunt $\geq0$. \\
Pentru a justifica ca un algoritm este util, aceste trebuie insotit de o justificare
ca solutia oferita este fezabila pentru problema, precum si o relatie intre ALG si OPT.
\quickdef{Acest tip de relatie este descrisa astfel:}{
Definitie 1: 
\begin{itemize}
\item un algoritm ALG pentru o problema de minimizare se numeste 
$\rho$-aproximativ, pentru o valoare $\rho >1$, daca ALG(I) $\leq \rho \cdot$OPT(I) pentru $\forall$ I - intrare \\
\item un algoritm ALG pentru o problema de maximizare se numeste 
$\rho$-aproximativ, pentru o valoare $\rho <1$, daca ALG(I) $\geq \rho \cdot$OPT(I) pentru $\forall$ I - intrare \\
\end{itemize}}
Observatie pentru problemele de minim: Orice algoritm $\rho$-aproximativ este la randul lui
$\rho$'-aproximativ pentru orice $\rho$' > $\rho$. De aceea, in cazul unul algoritm ALG pentru o problema de minimizare, spre exemplu, trebuie 
ca justificarea ce insoteste pe ALG sa ofere cea mai mică valoare $\rho$ pentru care ALG este $\rho$-aproximativ.
\quickdef{Definitie 2:}{ 
Fie ALG un algoritm $\rho$-aproximativ pentru o problema de minimizare. Spunem ca 
factorul de aproximare este "tight bounded" atunci cand avem $\rho$ = $supremum_I$
$\frac{ALG(I)}{OPT(I)}$. Ca sa aratam ca un algoritm este $\rho$-aproximativ "tight bounded",
trebuie deci sa justificam urmatoarele 2 lucruri:
\begin{itemize}
    \item Trebuie sa aratam ca este $\rho$-aproximativ, adica ALG(I)$\leq \rho \cdot$OPT(I) pentru orice intrare I
    \item Pentru orice $\rho$'<$\rho$ exista un I pentru care ALG(I)>$\rho$'$\cdot$OPT(I).
     Adesea totusi ne este la indemana sa aratam ca exista un I pentru care ALG(I) = $\rho \cdot$OPT(I)

\end{itemize}
}

Problema rucsacului 1/0\\
Enunt pe scurt: Trebuie sa gasim o submutime de obiecte de valoare totala maxima, fara ca greutatea lor totala sa depaseasca o capacitate data a rucsacului. Obiectele sunt puse integral in rucsac sau sunt data deoparte. Nu pot fi fractionate. 
In mod normal este programare dinamica cu complexiate n$\cdot$c. Presupunere: fiecare obiect are o greutate mai mica sau egala cu capacitatea rucsacului! 
Rezolvare propusa: \\
Fie L - lista obiectlor sortate dupa raportul $\frac{valoare}{greutate}$ \\
Fie $ O_p$ - obiectul cu profitul cel mai mare din lista de obiecte. \\
S = 0, G = capacitatea rucsacului; \\
Pentru fiecare O:L daca greutatea(O)$\leq$G, atunci S += val(O), G-= greutate(O) \\
ALG(I) = max(S, $O_p$) \\
Justificarea:
\begin{itemize}
    \item In primul rand, algoritmul de mai sus ne ofera o solutie fezabila. Elementele care au suma valorilor S vor avea o greutate totala $\geq$ capacitatea rucsacului, respectiv $O_p$ incape si el in rucsac de unul singur.
    \item Trebuie sa justificam doar factorul de aproximare. Fie $OPT_{1/0}$ valoarea optima pentru Problema Rucsacului in varianta 1/0, respectiv $OPT_G$ valoarea optima, furnizata de algoritmul de tip greedy pentru problema rucsacului in care avem voie sa fractionam obiecte pentru a le incarca in rucsac. 
    \item Cum este $OPT_{1/0}$ fata de $OPT_G$? \quad $OPT_{1/0} \geq OPT_G$
    \item Fie k indicele primului obiect care nu este adaugat in algoritmul de la inceputul paginii.
    \item $OPT_{1/0} \leq OPT_G \leq \sum_{1 \leq i \leq k} val(O_i) = \sum_{1 \leq i \leq k} val(O_i) + val(O_k) \leq \sum_{1 \leq i \leq k} val(O_i) + val(O_p)$
    \item ALG = max(S, $O_P$)
    \item $OPT_{1/0} \leq \sum_{1 \leq i \leq k} val(O_i) + val(O_p) \leq ALG + ALG = 2 \cdot ALG. OPT_{1/0} \leq 2 \cdot ALG$
    \item Ex intrare pt care abaterea e maxima: G = 100; Ob (val/greutate) = [(50+eps1)/(50+eps2), 50/50, 50/50] cu eps1 > eps2 > 0
    \item Profitul maxim este 100, iar profitul solutiei algoritmului este 50+eps1. ALG(I)$ \cong \frac{1}{2} \cdot OPT(I) deci \frac{1}{2}$ este un "tight upper bound" 
    \item 
% \end{definition}
\end{itemize}

Load Balancing Problem
Input:
\begin{itemize}
    \item m calculatoare identice; n activitati ce trebuiesc procesate. Fiecare activitate j avand nevoie de $t_j$ unitati de timp pentru executie.
    \item Odata initiata, fiecare dintre activitati trebuie derulata in mod continuu pe acelasi calculator.
    \item Un calculator poate executa cel mult o activitate in acelasi timp
\end{itemize}
Scop: Sa asignam fiecare activitate unui calculator astfel incat sa minimizam timpul pana cand toate activitatile sunt terminate.
%\textbf{Notatii:}
Notatii:
\begin{itemize}
    \item J(i) - submultimea tuturor activitatilor care au fost programate sa se desfasoare pe masina i.
    \item $L_i$ va reprezenta timpul de lucru al masinii i.
    \item $L_i = \sum_{j \in J(i)} t_j$
    \item Scop: O asignare a activitatile astfel incat $L_k$ este minimizat, unde k = $max_i(L_i)$, adica masina cu cel mai mare load.
    \item 
\end{itemize}
Pseudocodul:
\begin{itemize}[{}]
    \item Load - Balance(m, $t_1, t_2, ..., t_n$)
    \item for i = 1 to m:
    \begin{itemize}[{}]
        \item $L_i = 0; J(i) = \emptyset $  initializare: Fiecare Load este 0, iar multimea joburilor este nula pentru fiecare masina
    \end{itemize}
    \item for j = 1 to n:
    \begin{itemize}[{}]
        \item i = arg(min{$L_k | k \in$ {1,..., m}})  masina cu incarcatura cea mai mică in acest moment
        \item J(i) = J(i) $\cup$ {j}
        \item $L_i += t_j$ 
    \end{itemize}
\end{itemize}

In curs este step by step example, nu stiu daca sa il scriu \\

Care este factorul de aproximatie: \\

\quickdef{Lema 1}{OPT $\geq \max \{ \frac{1}{m} \cdot \sum_{1 \leq j \leq n} t_j, \max \{ t_j \mid 1 \leq j \leq n \} \}$}
\quickdef{Lema 2}{Algoritmul descris anterior este un algoritm 2-Aproximativ. Altfel spus, fie ALG = max($L_i | i \in [1,...,m])$ masina cea mai incarcata. Avem de aratat ca $ALG \leq 2 \cdot OPT$.}
Justificare: Fie:
\begin{itemize}
    \item k = indicele masinii cu load-ul maxim in urma rularii algoritmului
    \item q = ultimul job asignat masinii k
    \item load'(M) = load-ul masinii M dupa asignarea primelor q-1 joburi, dar nu si a jobului q
\end{itemize}
ALG = load'(k) t $t_q$ \\
load'(k) $ \leq \frac{1}{m} \sum_{1 \leq i \leq m}{} load'(i) = \frac{1}{m} \sum_{1 \leq j < q} t_j < \frac{1}{m} \sum_{1 \leq j \leq n} t_j \leq OPT$ (conforma lema 1)
$t_q \leq max{t_j | 1 \leq j \leq n} \leq OPT$ (conform lema 1)
ALG = load'(k) + $t_q \leq OPT + OPT = 2 \cdot OPT$

Se poate imbunatati Lower-Boundul? Avem 3 abordari: \\
\begin{itemize}
\item Acelasi algoritm, o analiza mai buna asupra lower-boundului folosit. \\
\quickdef{Teorema 1} {Algoritmul Greedy descris anterior este un algoritm 2-$\frac{1}{m} aproximativ$}
Justificare:    \\
load'(k) $ \leq \frac{1}{m} \sum_{1 \leq i \leq m}{} load'(i) = \frac{1}{m} \sum_{1 \leq j < q} t_j < \frac{1}{m}( \sum_{1 \leq j \leq n} t_j - t_q ) = \frac{1}{m}(\sum_{1 \leq j \leq n}t_j ) - \frac{1}{m} \cdot t_q $
ALG = load'(k) + $t_q \leq \frac{1}{m} \cdot (\sum_{1 \leq j \leq n} t_j) - \frac{1}{m} \cdot t_q + t_q = \frac{1}{m} \cdot (\sum_{1 \leq j \leq n} t_j) + t_q(1 - \frac{1}{m}) \leq OPT + OPT \cdot (1 - \frac{1}{m}) = (2 - \frac{1}{m}) \cdot OPT$
ALG $leq (2 - \frac{1}{m}) \cdot OPT$. Este tight bound? Da \\
Fie m masimi si m $\cdot$(m-1) activitati de cost 1 si o activitate de cost m. Algoritmul nostru va produce un load maxim de 2m-1 (o activitate de most m si m-1 activitati de cost 1). Solutia optima: o masina are 1 activitate de cost m. Restul de m-1 masini au cate m activitati de cost 1. Per total fiecare masina are un load de exact m unitati de timp. \\
ALG = 2m-1, OPT = m, ALG = (2-$\frac{1}{m}\cdot OPT)$\\
\item Acelasi algoritm, gasirea unui alt lower bound folosit alte inegalitati. \\

Fie o multime de n activitati cu timpul de procesare $t_1, t_2,...,t_n$ astfel incat $t_1 \geq t_2 \geq ... \geq t_n. Daca n>m, atunci OPT \geq t_m + t_{m+1}$
Daca n>m, atuni  $OPT \geq t_m + t_{m+1}$ \\
Justificare: Din principiul cutiilor rezulta faptul ca daca avem minim m+1 activitati si m masini, atuni cel putin o masina va avea cel putin 2 activitati. Fie acele activitati cu timpii de executie $t_i si t_j$ $OPT \geq t_m + t_{m+1}$
\item Un cu totul alt algoritm care poate da un total alt LB (Ordered-Scheduling Algorithm)
\begin{itemize}[]
    \item Fie algoritmul precedent la care adaugam urmatoarea preprocesare: activitatiile sunt sortate descrescator dupa timpul de lucru.
    \item \quickdef{Lema 3}{Fie o multime de n activitati cu timpul de procesare $t_1, t_2,...,t_n$ astfel incat $t_1 \geq t_2 \geq ... \geq t_n$. Daca n>m, atunci OPT $\geq t_m + t_{m+1}$} 
    \item \quickdef{Teorema 2}{Algoritmul descris este un algoritm $\frac{3}{2}$-aproximativ}
\end{itemize}
Justificari:
\begin{itemize}[]
    \item Fie k indicele masinii cu loadul maxim in urma executarii algoritmului, deci ALG = load(k)
    \item Fie q ultima activitate adaugata pe masina k
    \item Fie load'(i) = load-ul masinii fix inainte ca activitatea q sa fie asociata masinii k, adica load' semnifica loadul masiniilor dupa ce au fost distribuite primele q-1 activitati.
    \item ALG = load(k) = load'(k) + $t_q$
    \item load'(k) + $t_q \leq \frac{1}{m} \sum_{1 \leq j \leq n} t_j + t_q$.
    \item Ce se intampla daca q $\leq$ m? Atunci q va fi pusa peste o masina goala, deci ALG = $t_q \leq t_{max} (activitatea cu timpul de lucru maxim) \leq OPT$
    \item ALG = OPT - deci daca q este in primele m activitati, atunci algoritmul nostru este exact daca q>m. 
    \item Load'(k) + $t_q \leq \frac{1}{m} \sum_{1 \leq i \leq n} t_i + t_q < \frac{1}{m} \sum_{1 \leq i \leq n} t_i + \frac{1}{2}(t_m + t_{m+1}) \leq OPT + \frac{1}{2}OPT = \frac{3}{2}OPT$
\end{itemize}
\end{itemize}

Ciclu Hamiltonian (HC-Problem) \\

Fie G=(V, E) un graf neorientat. Numim ciclu hamiltonian un ciclu in G cu proprietatea ca fiecare nod apare exact o singură data. HC-Problem este problema de decizie daca intr-un graf oarecare exista sau nu un astfel de ciclu. HC-Problem este NP-Completa. \\

Travelling Salesman Problem (TSP) \\

Fie G un graf complet cu ponderi>0 pe muchii. Trebuie sa gasim ciclul hamiltonian de cost minim. Costul unui ciclu este suma costurilor muchiilor din componenta sa. TSP este o prblema NP-hard. Gasirea unui algoritm aproximativ este necesara. Dupa cum vom vedea, nu dispunem de un astfel de algoritm. \\
quickdef{Teorema 1}{
\begin{itemize}
    \item Nu exista nicio valoare c pentru care sa existe un algoritm in timp polinomial si care sa ofere o solutie cu un factor de aproximare c pentru TSP, decat daca P = NP.
    \item Demo: Vom arata ca exista un asemenea algoritm aproximativ, daca si numai daca putem rezolva problema HC in timp polinomial.
\end{itemize}}
Justificare: Reducere la absurd: Presupunem ca exista un algoritm aproximativ astfel incat daca exista S - costul optim al unui taseu inchis pentru TSP, atunci algoritmul nostru va oferi un traseu inchis de cost cel mult c$\cdot$S. \\
Fie G - un graf simplu, neponderat. Problema determimarii unui HC (Hamiltonian Cycle) in G este NPC. Construim un graf ponderat complet G' pe baza lui G dupa cum urmeaza:
\begin{itemize}
    \item V(G')=V(G); |V(G')| = n;
    \item Pentru fiecare e$\in$E(G') pentru care e$\in$E(G) avem w(e, G') = l (ponderea muchiei e in G' va fi 1).
    \item Pentru fiecare e$\in$E(G') pentru care e$\notin$E(G) avem w(e, G') = c$\cdot$n.
\end{itemize}
Daca G are un ciclu hamiltonian, solutia optina pentru TSP(G') va fi de cost n si va fi corespunzatoare ciclului hamiltonian din G. Costul maxim al solutiei date de algoritmul nostru aprximativ (cel care am presupus ca exista) ar fi cel mult n$\cdot$c. (1) \\
Daca G nu are un ciclu hamiltonian, atunci solutia optina pentru TSP(G) va fi de cost minim (n-1) + n$\cdot$c, deoarece trebuie sa se foloseasca cel putin o muchie $notin$E(G). Costul minim al solutiei date de algoritmul nostru aproximativ (cel care am presupus ca exista) este de cost > n$\cdot$c. (2)
\begin{itemize}
    \item graful G' se obtine in timp polinomial.
    \item Algoritmul ruleaza in timp polinomial.
    \item Din (1) si (2) rezulta faptul ca putem decide HCP in timp polinomial in functie de outputul algoritmului nostru pentru problema de TSP (daca outputul este cel mult n$\cdot$c, atunci graful G este Hamiltonian, altfel nu este Hamiltonian).
    \item Dar HCP este NPC din faptul ca P = NP, ceea ce este o contradictie.
\end{itemize}
Comis voiajorul de deplaseaza intr-un spatiu euclidian, deci respecta regula triunghiului (oricare 2 laturi adunate sunt mai mari decat a treia). Regula triunghiului pe grafuri: len(u,v) $\leq$ len(v, w) + len(w, u). Altfel spus, odata ce am traversat nodurile u, v, w - in aceasta ordine, este mai eficient ca sa ne intoarcem in u direct din w decat via v.
Obervatie 2: Fie G un graf complet, ponderat, care respecta regula triunghiului. Si fie $v_1, v_2, v_3,..., v_k$ un lant in graful G. Atunci avem len($v_1, v_k) \leq len(v_1, v_2, v_3,...,v_k).$ \\
Justificare: inductie
\begin{itemize}
\item presupunem ca len $(v_1, v_{k-1}) \leq len(v_1, v_2, v_3,...,v_{k-1})$
\item Din regula triunghiului avem ca $len(v_1, v_k) \leq len(v_1, v_{k-1}) \leq (v_1, v_2, v_3,...,v_{k-1})+len(v_{k-1}, v_k) = len(v_1, v_2, v_3,...,v_k)$
\end{itemize}
Travelling Salesman Problem (TSP) si Arbore Partial de Cost Minim (APCM) \\
Asemanari - cauta un traseu de cost total minim care sa cuprinda toate nodurile (ambele sunt conexe). Diferente - unul este arbore, altul este ciclu; una este P, iar alta este NP hard! \\
\quickdef{Lema 3}{Fie OPT costul solutiei optime pentru TSP, iar MST - ponderea totala a unui arbore partial de cost minim pe baza aceluiasi graf. Avem relatia $OPT \geq MST$ (OPT este ciclu, MST este arbore, deci OPT are 1 muchie in plus)} 
Justificare: Reducere la absurd: presupunem ca MST>OPT. Din ciclul Hamiltonian din care rezulta OPT putem scoate orice muchie si obtinem un lant. Cum lantul optinut are un costtotal mai mic decat al ciclului initial, dar lantul este un arbore, deci are costul cel putin egal cu MST, obtinem contradictia ca MST $leq$ OPT.
Algoritm approxTSP(G):
\begin{itemize}
    \item Calculam arborele partial de cost minim T pentru graful G.
    \item Algemen un nod u $\in$ T pe post de radacine si initializam $\Gamma$ gol.
    \item Parcurgere(u, $\Gamma$)
    \item Concatenam nodul u la finalul lui $\Gamma$ pentru a inchide un ciclu.
    \item return $\Gamma$
    \item 
\end{itemize}
Parcurgere(u, $\Gamma$)
\begin{itemize}
\item Concatenam pe u la $\Gamma$
\item Pentru fiecare v, fiu a lui u: Parcurgere(v, $\Gamma$)
\end{itemize}
///note - aici este un desen si exemple, nu le-am pus
Justificare: Observam ca ciclul nostru contine atat muchii/lanturi din MST - parcurse o singura data, dar si muchii care nu sunt in MST, dar costul unei muchii de forma (xy) - care nu este in MST - va fi mai mic decat costul unicului lant din MST care uneste x de y conform Lemei 2. ALG va contine muchii care sunt in MST si muchii care nu sunt in MST dar au cost mai mic decat lanturile echivalente din MST. $ALG \leq 2\cdot MST \leq 2\cdot OPT.$
Exista algoritmul lui Christofides, care este un algoritm $\frac{3}{2}$ aproximativ, care nu intra la examen.
